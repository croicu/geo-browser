# Stabilization

## Status: Ongoing

## Problem statement

On-device testing of the user layer and related features (image overlay, POI, zoom-out exit, geolocation dot) before starting new work. Goal is to collect and fix bugs found in the field, and tune constants that can only be validated on a real device (e.g. `PRESSURE_L_DELTA`, long-press timing, circle radius).

## Known open items

- `PRESSURE_L_DELTA = 10` — lightness shift per pressure unit; may need adjustment based on actual touch hardware range.
- Leaflet renderer guard (`geo_location.position_marker.renderer_not_ready`) — monitor whether the warning fires in practice and how often.
- Long-press threshold (600 ms) — validate feel on mobile; too short triggers accidentally, too long feels sluggish.

## Applied fixes

- **iOS long-press native menu / text selection** (`leafletFactories.ts` `createMap`): Added `user-select: none` + `-webkit-user-select: none` on the map container (cascades to zoom controls, preventing `+`/`−` glyph selection). Added capture-phase `contextmenu` listener on the container calling `e.preventDefault()` to block the iOS native popup when the touch lands on a Leaflet control rather than the map pane.
- **Heatmap layers frozen during a live pinch-zoom gesture** (`leafletFactories.ts`): regular circle-marker layers, the `__poi__` layer, and the precomputed `mundane`/`__void__` polygon layer all track a pinch gesture smoothly; heatmap layers stayed frozen at their pre-gesture position/scale for the whole gesture and only snapped to the correct place once fingers lifted. Root-caused by reading Leaflet 1.9.4's actual source (not guessing): a live touch pinch-zoom never fires the `'zoomanim'` event at all — `Map.TouchZoom._onTouchMove` drives the map via repeated `map._move()` calls instead, which only fire `'zoom'`/`'move'` (with a `#3530` comment forcing `'zoom'` to fire specifically for the pinch case). Leaflet's own SVG/Canvas path renderer listens for `'zoom'` for exactly this reason, which is why every SVG-rendered layer (circle markers, POI, the void polygon) already tracked correctly. `leaflet.heat`'s plugin only listens for `'zoomanim'` (repositions its canvas, cheap) and `'moveend'` (full data recompute, expensive) — never `'zoom'` — so during a pinch it does nothing at all until release. Fixed in `LeafletHeatLayerHandle.addTo()` by also listening for `'zoom'` and driving the plugin's own (cheap, CSS-transform-only) private `_animateZoom` method from it, mirroring what the SVG renderer does natively. Not unit-testable per CLAUDE.md's Leaflet-in-unit-tests ban — needs on-device confirmation. If this doesn't fully resolve it, the fallback is precomputing heatmaps at design time in `geo-builder` (matching the `__void__` layer's precedent) instead of patching the live `leaflet.heat` plugin further.
- **bbox outline not tappable inside its interior** (`leafletFactories.ts`, `contracts.ts`): `LayerFactory.createRectangle()` hardcoded `interactive: false` on every rectangle it created — fine for the three preview-only rectangles (`drawAreaInteraction.ts`'s drag preview, `bboxWidget.ts`'s edit-mode outline, `mapView.ts`'s pending-new-area preview, all of which want clicks to fall through to the map/drag handles underneath), but it also silently broke `AreaMarkerView`'s outline marker, which is supposed to be tap-to-jump like the circle marker. The existing unit test only exercised a stub rectangle that doesn't model real Leaflet hit-testing, so it never caught this. Added `interactive?: boolean` to `RectangleOptions` (default `true`), explicitly passed `interactive: false` at the three preview call sites, and `interactive: true` at `AreaMarkerView`'s. Regression tests added confirming both sides (`areaMarkerView.test.ts`, `bboxWidget.test.ts`).
- **"Ghost heatmap" — a peripheral area silently stealing current status while zooming out** (`areaRenderClassifier.ts`, `areaLifecycleTracker.ts`, `areaMarkerView.ts`): Reported 3× live via the same repro (click Berlin, zoom out, click Prague, zoom out) before being root-caused. Diagnosed from an actual browser console log (the newly-added `?logCategory` logging), not by guessing: zooming out from Prague dropped it below its own 48px threshold *and* simultaneously brought Berlin — physically larger, so it crosses its own 48px threshold at a much lower zoom — back above threshold, in the same `recompute()` tick. `AreaLifecycleTracker.selectCurrentArea()` only ever compared *candidates* (residency==="visible"), and Prague had just been excluded from candidacy by the same tick that made Berlin the sole candidate — so Berlin won by default, not by being closer to the viewport center (Prague was still centered; Berlin was at the edge). Root cause: the tracker had no equivalent of the old Summary/Detail split's `zoom < minZoom` exit boundary — a global zoom floor entirely separate from each area's own bbox-size threshold. Fixed with `AreaRenderClassifier.MIN_LOADED_ZOOM = 10`, as a genuinely separate axis from the bbox-size check, per the user's clarified experience spec: below this zoom, no area's layers ever show (forced to `outline` if bbox-big-enough, not `circle` — circle/outline is a pure bbox-size question, independent of zoom). Also corrected in the same pass, per the same clarification: `classifySize()` now compares the bbox's on-screen *area* against the area of a fixed 48px-diameter circle (not `max(width,height)` against 48px — a long thin strip could pass the old check while visually being far smaller than the circle it should have collapsed to), and `AreaMarkerView`'s circle marker is now a literal fixed-48px-diameter unfilled circumference (same color as the bbox outline), never rescaled on zoom — previously it scaled with the area's real-world radius in meters, shrinking as you zoomed out, which contradicted the "constant circle size" requirement entirely. See `tasks/layer_lifecycle.md`'s Rendering States section for the corrected spec. Regression tests in `areaLifecycleTracker.test.ts`/`areaRenderClassifier.test.ts` cover both the zoom-floor and area-vs-circle behavior. **Not yet verified on-device** — `MIN_LOADED_ZOOM = 10` is a starting guess (matches the old summary-viewport-zoom-≤10 clamp already in the codebase) and needs the same on-device check as `LOAD_THRESHOLD_PX`.
